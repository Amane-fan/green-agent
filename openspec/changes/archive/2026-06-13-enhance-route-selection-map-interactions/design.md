## Context

前端当前在同一个 React Leaflet 地图中渲染场景图边、标记和已规划路线。已规划路线会列在右侧面板中，但这些卡片目前只用于展示信息，地图也会一次性渲染所有已规划路线。自定义节点创建同样使用基于当前地图中心生成的坐标偏移，而不是用户真正想要的精确位置。

近期路线渲染改动已经基于 `route.path_node_ids` 计算路线渲染坐标，并包含重叠路线几何的弯曲位置处理。本次改动应复用该路线渲染路径，但将范围限制为选中的路线。不需要修改后端 schema，因为图边权重、车辆燃油率、路线路径和聚合路线指标已经存在于现有场景和规划 payload 中。

## Goals / Non-Goals

**Goals:**

- 让右侧路线卡片可被选择，并在前端状态中保存当前选中路线。
- 每次加载新的规划结果时，默认选中返回的第一条已规划路线。
- 在图边、标记和路线覆盖层下方渲染 React Leaflet 瓦片背景。
- 仅将选中的路线渲染为高亮路线覆盖层。
- 允许点击高亮的选中路线段，查看路段长度、所选车辆燃油率、预计路段燃油和预计路段碳排放。
- 保持基础图边可见且可点击，但仅用于查看长度。
- 将自动自定义节点放置替换为待放置流程，使用下一次地图点击位置作为节点坐标。

**Non-Goals:**

- 不修改后端路线规划、车辆分配、图验证或 API schemas。
- 不支持编辑已有节点位置。
- 不向后端规划响应添加逐路段指标。
- 不让未选中的路线在地图上可交互。

## Decisions

### 选中路线状态仅保留在前端

选中路线属于展示状态，SHALL 存储在 React app 中，不持久化到后端。当 `plan.routes` 发生变化时，如果存在路线，前端选中 `plan.routes[0].vehicle_id`；如果不存在路线，则清空选择。

考虑过的替代方案：将选择状态存储在后端或 URL 中。这样会增加持久化复杂度，但对本地演示流程没有实际价值。

### 高亮路线按路段 polylines 渲染

选中路线覆盖层应拆分为独立的图边路段，而不是一整条长 polyline。路段级渲染可以让点击处理和弹窗更精确：每个高亮路段都可以从匹配的图边推导自身长度，并使用所选车辆的 `fuel_per_km` 计算燃油。

考虑过的替代方案：保留一条路线 polyline，并根据点击坐标推断最近的路段。这需要额外的空间命中测试，可靠性也更低。

### 路段指标由客户端推导

对于选中路线段，前端推导：

- 长度来自匹配的无向图边权重；
- 燃油来自 `edge.weight * selectedVehicle.fuel_per_km`；
- 碳排放来自推导燃油乘以后端当前使用的同一碳排放系数。

路线级 `estimated_fuel` 仍作为总量的权威值；路段指标则是基于现有图和车辆数据的展示估算值。

考虑过的替代方案：向 `VehicleRoute` 添加后端逐路段指标。这会更显式，但会为了客户端可确定性推导的数据引入 API、model 和测试改动。

### 基础图边和高亮路线边使用不同弹窗

基础图边保持可见，用于保留地图道路骨架，并且只展示长度。选中路线图层位于其上方，使用更醒目的样式和路线专属弹窗。未选中路线段不会作为路线覆盖层渲染，因此不能暴露车辆燃油指标。

### 自定义节点创建使用待放置模式

点击“添加垃圾桶”、“添加车辆”或“添加处理厂”会设置待放置节点类型并显示提示。下一次地图点击会在点击的纬度/经度位置创建该节点，然后清除待放置状态。这样既保留当前简单的节点默认值，又让放置位置变得明确。

考虑过的替代方案：打开坐标输入 modal。直接点击地图更符合地图编辑流程，也避免手动输入坐标。

## Risks / Trade-offs

- [瓦片加载依赖外部 OSM 可用性] → 即使瓦片加载失败，也保持图边、标记和路线图层可用。
- [路线路径可能包含不存在于 `scenario.edges` 的图段] → 回退到几何路段长度估算用于展示；如果找不到图边权重，则只展示路线几何而不展示路段燃油。
- [同一条选中路线内的重叠高亮路线段仍可能重叠] → 推导路段位置时复用现有弯曲路线渲染。
- [待放置模式可能与地图平移冲突] → 仅在用户明确选择节点类型后，通过清晰的地图点击创建节点；创建后清除待放置模式。
- [碳排放系数重复定义可能与后端漂移] → 使用当前后端值定义本地常量，并在后端暴露逐路段指标前仅隔离用于路段展示。
