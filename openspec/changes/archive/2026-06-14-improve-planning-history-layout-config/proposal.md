## Why

当前首页把控制、地图、规划结果和规划历史全部挤在同一屏，历史记录占用控制面板空间，规划结果列表较长时会带动整个页面上下滚动，影响演示和对比不同规划结果。数据库连接也需要通过 shell 环境变量手动设置，不利于本地持久化配置复用。

本变更将历史查看、结果命名、数据库配置和首页布局一起收敛，目标是在桌面演示尺寸下保持首页主体稳定展示，并让用户能更清晰地管理和恢复不同规划结果。

## What Changes

- 规划记录增加独立的用户可编辑名称，默认名称由规划时上下文生成，历史页卡片支持内联重命名。
- 后端启动时从 `.env` 加载数据库连接配置，仓库提供 `.env.example`，真实 `.env` 继续不提交。
- 规划历史从首页左侧面板迁移到独立历史页面，首页只保留进入历史页的入口和当前规划结果展示。
- 点击历史页中的某条历史卡片后跳转到首页，并恢复该记录的场景和规划结果。
- 首页布局按 `1440x900` 优化，并在 `1366x768` 下避免页面主体整体上下滚动；需要滚动的内容限制在模块内部。
- 规划结果模块改为内部滑动窗口查看路线卡片，而不是拖动整个页面。
- 无 **BREAKING** API 移除；现有历史列表/恢复接口保持可用，并新增或扩展接口支持重命名。

## Capabilities

### New Capabilities

- 无

### Modified Capabilities

- `planning-history`: 增加规划记录可编辑名称、重命名接口行为和摘要/详情返回字段。
- `knowledge-sharing-pool`: 明确后端应从 `.env` 加载数据库连接配置，并继续保留 SQLite fallback。
- `route-visualization`: 调整历史查看为独立页面，定义历史卡片跳转恢复、首页单屏布局和规划结果内部滚动行为。

## Impact

- 后端模型与存储：`PlanningRecordSummary`、规划记录表结构、记录创建/查询/重命名逻辑。
- 后端 API：新增或扩展规划记录重命名接口；现有列表、详情、恢复接口需返回记录名称。
- 配置与依赖：引入 `.env` 加载方式，更新 `pyproject.toml`、`uv.lock`、`.env.example`、README 或演示文档。
- 前端 API 与类型：规划记录摘要类型增加名称字段，新增重命名请求封装。
- 前端页面：拆分首页和历史页导航，更新历史卡片、恢复流程、规划结果面板和整体 CSS 布局。
- 测试：覆盖后端 `.env` 配置加载、规划记录重命名、前端历史页跳转恢复、内联重命名和单屏布局样式约束。
