## Context

当前前端是一个 `App.tsx` 单页组件，左侧控制面板包含场景控制、自定义图编辑、统计指标和规划历史，中间是地图，右侧是规划结果。历史卡片点击后直接调用恢复接口并在当前页面展示。这个结构能完成演示，但随着历史记录和路线卡片增多，左侧和右侧内容会撑高页面，用户需要拖动整个页面查看结果。

后端规划记录已经持久化到 `planning_records`，但摘要只暴露 `scenario_name`，没有独立的用户可编辑名称。数据库连接目前直接读取 `GREEN_AGENT_DB_PATH` 和 `GREEN_AGENT_DATABASE_URL` 环境变量，README 通过 `export` 示例说明 MySQL 连接；仓库已经忽略 `.env` 和 `.env.*`，允许提交 `.env.example`。

## Goals / Non-Goals

**Goals:**

- 规划记录拥有独立的 `title`，支持在历史页卡片内联重命名。
- 后端启动时加载项目根目录 `.env`，让数据库连接配置可以写入本地配置文件。
- 历史记录迁移到 `/history` 页面，首页保留进入历史页的入口。
- 历史卡片点击后跳转首页，并恢复该记录对应的场景和规划结果。
- 首页按 `1440x900` 优化，在 `1366x768` 下避免主体整体上下滚动；长内容在模块内部滚动。
- 规划结果列表使用内部滑动窗口查看路线卡片。

**Non-Goals:**

- 不新增规划记录删除、搜索、筛选或批量管理。
- 不改路线规划算法、地图渲染语义或历史记录保存时机。
- 不把真实 `.env` 提交到仓库，不在文档中要求提交数据库密码。
- 不引入完整前端路由框架；当前只需要首页和历史页两个视图。

## Decisions

### 1. 使用 `planning_records.title` 作为规划结果名称

新增 `title` 字段保存用户可编辑的规划结果名称，默认值由后端在保存规划记录时生成，例如使用场景名、模拟时间和记录创建时间组合。`scenario_name` 保持为规划时场景快照元数据，不承担用户命名职责。

备选方案是复用 `scenario_name`。该方案会把“场景名称”和“某次规划结果名称”混在一起，用户重命名历史记录后会误以为场景本身被改名，因此不采用。

### 2. 新增 `PATCH /api/planning-records/{record_id}` 做重命名

请求体使用 `{ "title": "..." }`，后端 trim 后校验非空和最大长度，返回更新后的 `PlanningRecordSummary`。列表、详情和恢复接口都返回包含 `title` 的摘要。前端历史页内联编辑时调用该接口，成功后只更新对应卡片，不触发恢复。

备选方案是把重命名合并到详情接口或恢复接口。该方案会让读取/恢复产生写入副作用，不利于测试和理解，因此不采用。

### 3. 通过 `python-dotenv` 在应用初始化前加载 `.env`

后端在读取 `os.getenv` 前调用 `load_dotenv()`。`.env.example` 提供 `GREEN_AGENT_DATABASE_URL` 和 `GREEN_AGENT_DB_PATH` 示例；真实 `.env` 继续由 `.gitignore` 忽略。未配置 MySQL 时继续使用 SQLite fallback，测试也应保持不依赖 MySQL。

备选方案是手写 `.env` 解析器。该方案容易遗漏转义、引号和注释规则，维护价值低，因此采用成熟依赖。

### 4. 前端使用轻量路径状态区分首页和历史页

不引入 `react-router`。`App` 根据 `window.location.pathname` 渲染首页或历史页，并监听 `popstate` 处理浏览器返回。历史页链接使用 `/history`，历史卡片点击跳转到 `/?record_id=<id>`。首页加载或路径变化时，如果存在 `record_id` 查询参数，则调用恢复接口并展示对应场景和规划结果。

备选方案是引入 `react-router`。当前只有两个页面，引入路由库会增加依赖和测试改造成本，收益不足。

### 5. 首页与历史页拆分组件边界

将当前 `App.tsx` 中的历史列表逻辑拆成历史页组件，将首页控制区、地图区、规划结果区保持在首页。规划状态继续由顶层 `App` 管理，历史页只负责列表、重命名和跳转。这样可以减少单个面板的职责，也方便测试历史页交互。

### 6. 单屏布局使用视口高度和模块内部滚动

首页外层使用接近 `100vh` 的应用布局，顶部 hero 压缩高度，主体使用固定高度网格。左侧控制面板和右侧规划结果面板使用 `overflow: hidden` 包裹内部滚动区；路线列表放入独立滚动容器。地图容器填满中间列高度。目标是在 `1440x900` 下完整展示主要模块，在 `1366x768` 下页面主体不整体上下滚动。

备选方案是继续依赖文档流自然撑开。该方案实现简单，但无法满足“尽量一个页面展示完全”和“规划结果滑动窗口”的目标。

## Risks / Trade-offs

- [Risk] 现有 SQLite 或 MySQL 表没有 `title` 列 → 启动初始化时检测列并执行兼容迁移，旧记录用 `scenario_name` 和记录 ID/时间回填默认标题。
- [Risk] MySQL `ALTER TABLE ADD COLUMN IF NOT EXISTS` 兼容性差异 → 使用查询列信息后按需执行普通 `ALTER TABLE`，分别处理 SQLite `PRAGMA table_info` 和 MySQL `SHOW COLUMNS`。
- [Risk] 查询参数恢复可能与初始场景加载并发 → 首页初始化时优先处理 `record_id`；没有 `record_id` 时再加载当前场景。
- [Risk] 历史卡片内联编辑和卡片点击恢复冲突 → 重命名控件阻止事件冒泡；只有非编辑区域点击才跳转恢复。
- [Risk] 固定视口布局在更小屏幕上拥挤 → 目标尺寸限定为 `1440x900` 和 `1366x768`；小于该尺寸时允许模块内部滚动或横向最小宽度保持现状。
- [Risk] `.env` 中的真实凭据泄露 → 保持 `.env` / `.env.*` 忽略规则，只提交 `.env.example`，文档明确不要提交真实配置。

## Migration Plan

1. 添加 `title` 字段到后端模型、SQLite/MySQL schema 和初始化迁移逻辑。
2. 保存新规划记录时生成默认 `title`，列表、详情、恢复接口返回 `title`。
3. 新增 `PlanningRecordRenameRequest` 和 `PATCH /api/planning-records/{record_id}`。
4. 引入 `python-dotenv` 并在应用初始化前加载 `.env`，补充 `.env.example` 和文档。
5. 前端类型和 API 封装增加 `title` 与重命名请求。
6. 拆分首页和历史页，使用轻量路径状态和 `record_id` 查询参数恢复记录。
7. 调整 CSS 为固定视口布局和规划结果内部滚动窗口。
8. 更新后端、前端和规格相关测试。

回滚策略：如果迁移后需要回退代码，保留数据库中的 `title` 列不会影响旧代码；旧代码会忽略该列。取消 `.env` 后仍可通过 shell 环境变量或 SQLite fallback 启动。
