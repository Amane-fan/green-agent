## Context

本项目是 SE2026-12“绿运先锋——垃圾分类收运多智能体路径优化系统”的 MVP。仓库当前只有 OpenSpec 配置，没有既有应用代码，因此本变更会同时建立后端、前端、算法和演示数据能力。

系统需要在 React Leaflet 地图上表达用户创建或随机生成的垃圾桶、收运车辆、处理厂和道路边，并将这些地图元素抽象为无向连通加权图。后端负责图校验、满溢模拟、知识共享池、LangGraph 多智能体编排、可达性分析和路径优化；前端负责地图编辑、时间模拟交互、路线展示和指标展示。

## Goals / Non-Goals

**Goals:**

- 建立一个可本地运行的 Python + uv 后端和 React + React Leaflet 前端。
- 支持自定义图和随机图两种场景创建方式。
- 随机图默认生成 30 个垃圾桶、5 辆收运车、3 个处理厂，并保证无向、连通、有边权。
- 按垃圾类别进行硬约束收运，车辆只能收运支持类别，处理厂只能接收支持类别。
- 使用 LangGraph 编排多 Agent 流程，但让图分析和路径优化保持确定性。
- 使用遗传算法作为默认路径规划方法，目标兼顾最短距离和高满溢垃圾桶优先。
- 展示每辆车的不同颜色路线、任务顺序、总距离、油耗和碳排放估算。

**Non-Goals:**

- 不接入真实道路导航、交通拥堵、地理编码或商业地图路径 API。
- 不实现真实 IoT 设备接入，垃圾桶满溢数据使用模拟数据。
- 不实现用户登录、多租户、权限管理或云端部署。
- 不追求工业级 VRP 全局最优解，MVP 以可解释、可演示、可测试为优先。
- 不要求车辆在规划后真实执行路线；路线执行和清运后的状态更新作为可选演示动作处理。

## Decisions

### 使用前后端分离的本地演示架构

后端使用 Python + uv + FastAPI，前端使用 React + Vite + React Leaflet。FastAPI 提供场景、模拟、规划和结果查询接口，React 前端只负责交互和展示。

替代方案是把算法全部放到前端运行，但这会让 LangGraph、多智能体编排、Python 路径优化和知识共享池难以体现。后端集中处理业务图和算法，更符合课题技术栈。

### 地图只是交互层，业务图由后端校验

React Leaflet 提供地图点位、边和路线的可视化，后端维护规范化图结构。每个节点包含 `id`、`type`、`lat`、`lng`、垃圾类别、容量、满溢率等属性；每条边包含 `source`、`target`、`weight`，并按无向边处理。

替代方案是依赖真实地图道路数据，但外部 API、网络环境和额度会影响演示稳定性。MVP 使用用户定义边和随机边权，保证演示可控。

### 随机图生成必须先保证连通性

随机场景先生成节点坐标，再用最小生成树或近似主干连接所有节点，随后添加若干近邻边形成更自然的路线选择空间。边权使用节点间距离或缩放后的地理距离。

替代方案是简单随机连边，但可能生成不连通图，导致演示前置条件不稳定。先构造连通主干可以保证规划入口可靠。

### 分类收运作为硬约束

垃圾桶、车辆和处理厂都携带垃圾类别信息。规划时先按类别筛选可服务车辆和可接收处理厂，类别不匹配的路线不进入可行解。

替代方案是把类别作为惩罚项，但这会允许“错收错运”的路线出现在结果中，不符合垃圾分类课题要求。

### LangGraph 编排确定性工具

LangGraph 的节点表示垃圾桶监测 Agent、知识共享池 Agent、任务编排器 Agent、可达性分析 Agent、车辆 Agent 和处理厂 Agent。每个 Agent 处理共享状态并调用确定性 Python 函数；路径优化、最短路、连通性和满溢率计算不依赖 LLM。

替代方案是让 LLM 做任务分配或路线解释。MVP 中关键计算必须可复现，LLM 可作为后续路线说明扩展，不作为规划核心。

### 默认路径优化采用遗传算法

遗传算法编码车辆任务分配和每辆车的访问顺序，适应度函数综合路线总距离、高满溢垃圾桶访问延迟、容量约束、类别约束和不可达惩罚。对不可行解采用硬过滤或高惩罚，最终输出可解释路线。

替代方案是蚁群算法。蚁群算法也适合路径搜索，但多车辆、分类约束和容量约束下编码与调参更复杂；遗传算法更容易在课程项目中解释染色体、适应度和交叉变异。

### 知识共享池使用 SQLite 或本地文件持久化

MVP 推荐 SQLite 存储场景状态、垃圾桶满溢历史、预测结果和规划记录。它适合本地演示，便于查询，也不需要额外服务。

替代方案是内存存储或 Postgres。内存存储重启后丢失演示历史；Postgres 对本地课程项目部署成本偏高。

## Risks / Trade-offs

- [随机图路线不自然] → 使用近邻边和距离权重，避免大量跨地图长边。
- [遗传算法结果不稳定] → 支持固定 `seed`、限制迭代次数，并在结果中展示启发式性质。
- [30 个垃圾桶导致前端路线拥挤] → 使用图层开关、路线高亮和车辆颜色区分，避免一次性突出所有辅助信息。
- [车辆或处理厂类别覆盖不足] → 随机生成阶段必须保证车辆能力和处理厂接收能力覆盖所有已生成垃圾类别。
- [LangGraph 被误用为算法实现] → Agent 只负责状态编排，算法放在独立可测试模块。
- [真实地图距离和边权不一致] → 明确边权代表用户定义的路径长度，不承诺真实道路距离。

## Migration Plan

这是新项目初始化，无既有数据迁移。实现时先创建后端和前端骨架，再逐步接入场景管理、模拟、规划和展示能力。若某个模块失败，可以保留静态示例场景作为演示回退。

## Open Questions

- 满溢率达到多少时进入待收运任务，MVP 默认使用 70%，前端应允许调整。
- 车辆容量单位使用抽象容量值还是垃圾桶容量总和，MVP 默认使用抽象单位。
- 处理厂是否有每日处理容量，MVP 可先记录容量字段，但规划只校验垃圾类别。
