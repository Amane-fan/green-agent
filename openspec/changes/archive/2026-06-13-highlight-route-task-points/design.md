## Context

前端当前已经支持在 React Leaflet 地图中只高亮当前选中的车辆路线，并通过 `route.path_node_ids` 展开路线几何路径。规划结果同时提供 `route.stops`，其中包含本次实际回收的垃圾桶和目标处理厂；车辆起点可以通过 `route.vehicle_id` 在 `scenario.vehicles` 中找到对应 `node_id`。

现在的问题是，完整路径可能经过一些不需要回收的垃圾桶。地图上所有垃圾桶 marker 视觉层级相近，用户会误以为路径经过的垃圾桶都需要收运；右侧路线卡片也没有显示车辆起点，路线任务链不完整。

## Goals / Non-Goals

**Goals:**

- 让选中路线的车辆起点在地图和右侧路线卡片中清晰可见。
- 让选中路线中实际需要回收的垃圾桶比普通途经垃圾桶更突出。
- 保持完整路线几何路径可见，但不把 `path_node_ids` 中的途经点当成任务点展示。
- 复用现有规划响应，不修改后端 API schema。
- 为前端派生逻辑和可视化输出增加测试覆盖。

**Non-Goals:**

- 不改变路线规划算法、车辆分配、最短路径展开或后端 models。
- 不把所有途经节点加入右侧任务序列。
- 不为未选中路线展示任务点高亮。
- 不新增地图编辑能力。

## Decisions

### 任务点来源以 `route.stops` 为准

选中路线的“需要回收垃圾桶”只从 `selectedRoute.stops` 中 `node_type === "bin"` 的记录派生。`path_node_ids` 继续只用于绘制车辆实际行驶路径和路段弹窗，不参与判断某个垃圾桶是否需要收运。

替代方案是把 `path_node_ids` 中所有 bin 节点都显示在右侧，并标注“经过/回收”。这会暴露更多底层路径细节，但会让右侧面板再次变长并稀释任务重点，因此不采用。

### 车辆起点由前端从场景中派生

车辆起点不进入后端 `VehicleRoute.stops`，前端通过 `selectedRoute.vehicle_id` 匹配 `scenario.vehicles[].id`，再读取对应 `vehicle.node_id` 和节点坐标。右侧任务序列在回收点之前显示这个起点，地图上对该车辆节点应用选中路线起点高亮。

替代方案是修改后端，让 `stops` 包含起点。这样可以让响应更显式，但会改变现有 `RouteStop.node_type` 语义和 API 契约；当前需求可由已有数据可靠派生，不需要扩展 schema。

### 右侧路线卡片展示任务链而非几何路径

路线卡片继续作为选中路线入口，但内部展示从“节点 id 串联文本”升级为结构化任务链：车辆起点、按 `stop.order` 排序的回收垃圾桶、目标处理厂。回收点显示顺序号和可用的满溢率；目标处理厂显示为终点。

替代方案是保留现有单行 `node_id -> node_id` 文本，并只在地图做高亮。这样改动更小，但无法解决右侧面板缺少车辆起点和任务重点不够清晰的问题。

### 地图 marker 使用叠加高亮而不改变基础节点含义

基础节点 marker 仍按当前节点类型、垃圾类别和满溢率渲染。当前选中路线额外派生起点和回收点集合，对这些节点应用更强的视觉样式，例如更大半径、路线色外环、编号或专用 class。非任务途经垃圾桶保持普通 marker。

替代方案是隐藏非任务途经垃圾桶。这样会更突出任务点，但会破坏场景全图可读性，也不利于理解路线为什么绕行。

## Risks / Trade-offs

- [车辆或节点数据不完整] → 如果无法从 `vehicle_id` 找到车辆节点，则右侧省略起点高亮并保持路线卡片可用；地图不渲染缺失节点的起点强调。
- [路线中同一垃圾桶重复出现] → 任务点集合按 `route.stops` 去重用于地图高亮，右侧仍按 `stop.order` 展示实际任务顺序。
- [marker 编号在地图缩放较小时拥挤] → 仅当前选中路线的任务点显示编号，并控制 marker 尺寸与文字层级。
- [视觉层级过强遮挡基础节点信息] → 通过 CSS class 和外环表达选中状态，Popup 仍保留节点 id、类型、类别和满溢率。
