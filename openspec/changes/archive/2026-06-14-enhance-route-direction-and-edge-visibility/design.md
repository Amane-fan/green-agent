## Context

前端当前使用 React Leaflet 在同一张地图中渲染 OpenStreetMap 瓦片、场景节点、基础图边和当前选中车辆路线。基础图边来自 `scenario.edges`，业务语义是无向加权图；选中路线来自 `route.path_node_ids`，具有明确的节点顺序。

现有选中路线已经按路段拆分为多个 `Polyline`，用于保持路段级弹窗和重叠路线偏移能力。普通基础图边当前样式较细且透明度较低，在瓦片背景下不够明显。

## Goals / Non-Goals

**Goals:**

- 使用流动虚线表达当前选中车辆路线的行驶方向。
- 让流动方向遵循 `path_node_ids[index] -> path_node_ids[index + 1]` 的顺序。
- 增强普通基础图边在瓦片背景下的可见性。
- 保持普通基础图边的无向语义，不在基础图边上显示方向效果。
- 保留现有路线选择、选中路线段弹窗和任务点高亮行为。

**Non-Goals:**

- 不修改后端规划算法、API schema、数据模型或路线响应结构。
- 不为未选中路线渲染路线覆盖层。
- 不引入新的地图渲染依赖。
- 不把基础图边改成有向边。
- 不改变路线段指标的计算方式。

## Decisions

### 选中路线使用分段流动虚线

选中路线继续按路段渲染，每个路段使用路线颜色和 `selected-route-segment` 样式，并通过 CSS `stroke-dasharray` 与 `stroke-dashoffset` 动画形成流动虚线。这样可以复用现有路段拆分、重叠偏移和弹窗逻辑。

考虑过的替代方案是额外渲染一条整路线装饰层，再保留分段透明点击层。它能让整条路线的虚线看起来更连续，但会增加图层和测试复杂度。当前需求重点是方向可读性，分段流动已经足够明确。

### 流动方向由路径点顺序决定

每条高亮路段的坐标顺序保持 `source -> target`，其中 `source` 和 `target` 直接来自 `path_node_ids` 的相邻节点。CSS 动画方向必须与 SVG path 的坐标方向一致，使虚线从前序节点流向后序节点。

考虑过的替代方案是在每条线段中加入独立箭头 marker。箭头更静态，也容易与用户要求的“流动虚线”不一致，因此不采用。

### 基础图边只增强可见性，不表达方向

普通基础图边属于场景图骨架，业务上是无向边。它们应使用更清晰的颜色、线宽和透明度，必要时使用独立 class 方便样式和测试识别，但不能添加流动虚线或箭头。

考虑过的替代方案是所有图边都使用虚线。这样会把普通边和路线方向混在一起，也会暗示基础图边有方向，因此不采用。

### 测试覆盖样式契约而非动画帧

单元测试应验证选中路线段带有方向样式 class、基础图边样式更明显、基础图边不带方向 class，并保持未选中路线不渲染覆盖层。动画实际运行属于浏览器 CSS 行为，不在 Vitest 中逐帧验证。

## Risks / Trade-offs

- [CSS 动画在测试环境中无法真实播放] -> 通过 class、路径顺序和样式属性验证契约，必要时由人工或浏览器测试确认视觉效果。
- [分段虚线在节点连接处不是完全连续] -> 保留现有路段级弹窗和重叠偏移能力，以方向可读性优先。
- [过强的基础边样式可能与选中路线竞争视觉层级] -> 基础边仅增强到可辨识，选中路线仍使用路线色、更粗线宽和流动效果建立更高层级。
- [动画可能对部分用户造成干扰] -> 动画仅应用于当前选中路线，数量有限；后续如需无障碍优化，可增加 `prefers-reduced-motion` 降级样式。
