## Context

前端是一个桌面优先的调度工作台，主要由顶部状态区、左侧控制面板、中间 React Leaflet 地图、右侧规划结果面板、历史页和浮动 AI 助手组成。当前样式集中在 `frontend/src/styles.css`，并使用深色背景、半透明面板和绿色高亮。用户已确认采用“清爽白绿工作台”方向，并明确要求不改变布局和交互。

## Goals / Non-Goals

**Goals:**

- 将现有深色视觉刷新为亮色视觉，主基调为白色、浅绿和柔和中性色。
- 保留绿色环保识别，让主操作、选中态和业务强调色保持一致。
- 保证首页、历史页、路线卡片、表单控件、地图容器和 AI 助手在亮色背景下具有足够对比度。
- 保持现有页面结构、三栏布局、滚动区域、按钮行为、地图交互和 AI 助手交互不变。

**Non-Goals:**

- 不新增主题切换、用户偏好保存或暗色/亮色双主题机制。
- 不重构组件结构，不拆分 JSX，不改变路由、状态管理或 API 调用。
- 不调整地图瓦片源、路线规划逻辑、历史记录逻辑或 AI 助手请求逻辑。
- 不引入新的 CSS 框架、设计系统依赖或图标库。

## Decisions

### Decision: 以 `styles.css` 为主要改动点

将颜色、背景、边框、阴影、表单和状态样式集中在 `frontend/src/styles.css` 中调整。这样可以保留 React 组件结构和交互逻辑，符合“只换肤”的范围。

替代方案是引入 CSS 变量和主题层，但当前项目只有单一主题且样式文件规模可控。为一次性亮色刷新引入主题系统会扩大范围，并产生后续维护成本。

### Decision: 保留绿色为主强调色

主按钮、选中态、状态强调和部分提示信息继续使用绿色系，但从深色荧光绿改为适合白底的绿色，例如深绿文本、浅绿背景和中等饱和度边框。这样既延续“绿运先锋”的环保业务语义，又避免亮色界面中出现刺眼的高饱和色块。

替代方案是改成蓝色科技风或浅灰运营风。探索阶段已选择白绿方向，因此蓝色只适合作为地图、可回收类别或辅助信息色，不应成为主视觉。

### Decision: 使用实体浅色面板替代深色玻璃面板

顶部、控制区、规划结果区、历史页和 AI 助手应使用白色或接近白色的实体面板，配合浅绿/浅灰边框和柔和阴影。深色半透明玻璃效果在亮色主题下会造成脏灰感，并降低表单和路线卡片的可读性。

替代方案是保留玻璃质感但提高透明度。该方案对地图区域和复杂卡片不稳定，且容易和 Leaflet 瓦片背景混杂。

### Decision: 地图相关颜色只做可读性适配

React Leaflet 瓦片源、节点类型色和路线颜色语义保持不变。若现有基础图边颜色在亮色界面或地图瓦片上不清晰，可以小幅调整 `App.tsx` 中直接写死的边线颜色，或在 CSS 中调整容器背景，但不改变路线选择、流动虚线和 popup 行为。

替代方案是重新定义所有地图色板。该做法会触及已有 `route-visualization` 语义，超出本次纯视觉主题刷新范围。

## Risks / Trade-offs

- 亮色背景降低浅色路线或浅色文本的可见性 → 对路线卡片、选中态、表单、禁用按钮和错误提示分别检查对比度，并对浅色文本改用深色文本。
- 全局颜色替换遗漏 AI 助手或历史页局部样式 → 按样式类覆盖范围逐段检查 `planning-assistant-*`、`history-*` 和 `route-*` 选择器。
- 地图瓦片本身是亮色，外层容器过亮可能弱化边界 → 为地图容器保留清晰边框和轻微阴影，基础图边使用适中深度的中性色。
- 只改 CSS 可能留下少量 JSX 内联颜色不适配 → 限制在必要的地图边线颜色调整，不碰组件结构和业务逻辑。

## Migration Plan

1. 更新 `frontend/src/styles.css` 的全局背景、面板、表单、按钮、卡片、提示、历史页和 AI 助手颜色。
2. 检查 `frontend/src/App.tsx` 中地图基础图边的内联颜色，必要时改为适配亮色界面的中性色。
3. 运行前端测试和构建，确认没有行为回归。
4. 手动查看首页、历史页和 AI 助手弹层，确认亮色主题覆盖完整。

回滚策略是撤销本次样式改动；由于不涉及数据结构、API 或依赖，回滚不需要迁移步骤。
