01 / 问题背景

分类收运的核心难点是“多约束路径规划”

  • 分类约束：车辆、垃圾桶、处理厂必须类别匹配，不能混收。
  • 时序约束：满溢率持续变化，需要提前识别即将超阈值的点位。
  • 路径约束：道路图必须可达，访问顺序直接影响里程、油耗和碳排放。
讲述重点：这是车辆路径问题，但比普通最短路多了分类、容量、可达性和满溢优先级。
输入

垃圾桶位置、满溢率、容量、垃圾类别、道路边权、车辆能力、处理厂接收范围。

处理

Agent 编排任务，筛选待收运垃圾桶，做可达性分析并规划每辆车路线。

输出

车辆收运顺序、完整路径节点、距离、油耗、碳排放和未分配原因。

02 / 系统架构

前端可视化 + 后端规划服务 + 知识共享池

技术栈与模块

  • React + React Leaflet：展示地图点位、道路边和车辆路线。
  • FastAPI：提供场景生成、时间推进、路线规划和历史记录 API。
  • LangGraph：串联监测、知识、编排、可达性、车辆处理厂匹配 Agent。
  • NetworkX + KnowledgePool：完成图计算，并沉淀场景、预测和规划历史。
讲述重点：架构不是单纯调用算法，而是把感知、预测、任务拆分、路线优化和历史数据沉淀成闭环。

03 / 多智能体任务编排

LangGraph 将规划流程拆成 5 个 Agent 节点

Agent 1 Monitoring 预测未来 3 步满溢率，筛选超阈值垃圾桶。
Agent 2 Knowledge 加载场景、预测和历史记录，提供共享上下文。
Agent 3 Task Orchestrator 按垃圾类别组织待收运任务。
Agent 4 Reachability 基于道路图过滤不可达点位。
Agent 5 Vehicle Facility 完成车辆分配、路径排序和处理厂选择。
讲述重点：每一步都会写入 trace，前端可以解释“为什么这些垃圾桶被收运、为什么有些未分配”。

04 / 路径规划实现 A

先做任务筛选与车辆分配，再做路线排序

  1. 预测与筛选：当前或未来 3 步满溢率超过阈值的垃圾桶进入待收运集合。
  2. 优先级：按满溢率从高到低处理，优先解决高风险点位。
  3. 候选车辆：类别匹配、路径可达、剩余容量足够才允许分配。
  4. 分配策略：优先任务数更少、距离更近的车辆，兼顾负载均衡。
候选车辆条件： vehicle.supported_waste_type == bin.waste_type has_path(vehicle.node_id, bin.id) has_path(bin.id, compatible_facility.node_id) remaining_capacity >= estimated_volume
estimated_volume = bin.capacity × fill_rate / 100 vehicle = min(任务数量, 到垃圾桶距离, 随机扰动)

05 / 路径规划实现 B

重点：用遗传算法优化单车收运顺序

  • 当某辆车分配到超过 2 个垃圾桶时，进入遗传算法排序。
  • 初始种群包含贪心路径和随机排列，兼顾初始质量与搜索空间。
  • 每代保留优质父代，执行顺序交叉与两点交换变异。
  • 迭代 35 代后选择适应度最低的顺序作为收运顺序。
fitness = route_distance + delay_penalty × 0.01 delay_penalty = Σ((visit_order + 1) × fill_rate)

06 / 输出指标与可视化

从“收运顺序”还原为可展示的完整道路路径

  • 路径序列为：车辆当前位置 → 多个垃圾桶 → 兼容处理厂。
  • 相邻节点之间调用 NetworkX 最短路径，拼接成完整 `path_node_ids`。
  • 输出距离、油耗、碳排放，并解释不可达、容量不足或类别不匹配等未分配原因。

07 / 工程保障与演示价值

系统不仅能规划路线，还能解释和复现实验结果

可达性分析

分配前用 `nx.has_path` 检查车辆、垃圾桶、处理厂之间是否可达。

知识共享池

保存 `fill_history`、`predictions`、`planning_records`，支持复盘。

测试覆盖

覆盖类别约束、不可达任务、容量限制、高满溢优先级和 API 编排。

课堂演示

生成场景、推进时间、规划路线、查看历史记录，形成演示闭环。
