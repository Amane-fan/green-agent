01

01 / Problem

分类收运不是普通最短路，而是动态多约束调度

垃圾桶满溢率会随时间变化，车辆和处理厂又存在垃圾类别、容量和道路可达性约束。 如果只按最近距离收运，容易出现高满溢点位延迟、车辆混收或终点不可处理的问题。

汇报重点：我们把问题定义成“先判断哪些点该收，再判断谁能收，最后优化怎么走”。
约束一 分类匹配

垃圾桶、车辆、处理厂必须类别兼容，系统显式拒绝混收和错误处理终点。

约束二 动态满溢

不仅看当前 fill_rate，还预测未来 3 个时间步，提前纳入即将超阈值点位。

约束三 图可达性

分配前检查车辆到垃圾桶、垃圾桶到处理厂是否有路径，避免生成不可执行方案。

03 / Architecture

前端负责可视化，后端负责领域决策，知识池负责状态沉淀

前端层React、TypeScript、React Leaflet：地图点位、道路边、车辆路线、历史记录和 AI 聊天界面。
接口层FastAPI：提供 `/api/scenarios`、`/api/simulation`、`/api/planning`、`/api/planning-records`、`/api/ai-assistant`。
编排层LangGraph：将规划拆成 monitoring、knowledge、task_orchestrator、reachability、vehicle_facility。
算法层NetworkX 最短路 + 约束过滤 + 遗传算法，输出车辆任务顺序和完整 path_node_ids。
数据层KnowledgePool：保存场景、满溢历史、预测结果、事件和规划快照。
04

04 / Multi-Agent Workflow

LangGraph 把一次路线规划拆成 5 个可追踪节点

每个节点只处理一个清晰职责，并向 trace 写入解释信息。前端和 AI 助手可以基于 trace 回答“为什么这样规划”。

1 Monitoring

计算 horizon=3 的满溢预测，找出当前或未来超过阈值的垃圾桶。

2 Knowledge

加载当前场景与预测上下文，让后续节点共享同一份事实基础。

3 Task Orchestrator

按垃圾类别组织待收运任务，为车辆能力匹配做准备。

4 Reachability

基于道路图准备可达性分析，避免不可执行路线进入结果。

5 Vehicle Facility

完成车辆分配、路径排序、处理厂选择，并生成最终 PlanningResult。

workflow: START → monitoring → knowledge → task_orchestrator → reachability → vehicle_facility → END
05

05 / Route Planning

路线规划分三步：筛选任务、分配车辆、优化单车顺序

  1. 先用阈值与预测结果确定待收运垃圾桶。
  2. 再按类别、可达性、剩余容量筛选候选车辆。
  3. 每辆车的多个任务用遗传算法排序，高满溢点位获得更高访问优先级。
这里体现软件工程价值：算法不是孤立函数，而是和领域模型、图验证、API、前端解释共同组成闭环。 
候选车辆条件：
vehicle.supported_waste_type == bin.waste_type
has_path(vehicle.node_id, bin.id)
has_path(bin.id, compatible_facility.node_id)
remaining_capacity >= bin.capacity * fill_rate / 100
单车排序评分：
fitness = route_distance + delay_penalty * 0.01
delay_penalty = Σ((visit_order + 1) * fill_rate)
遗传算法参数

24 个初始个体，35 代迭代，每代保留 8 个父代并生成 16 个子代。

结果指标

输出 distance、estimated_fuel、estimated_carbon，并保留未分配原因。

06 / Genetic Algorithm

遗传算法负责优化单车收运顺序

当某辆车分配到多个垃圾桶时，系统不直接按距离贪心走完，而是把收运顺序作为染色体， 在类别、容量、可达性已经满足的前提下搜索更短路线，并让高满溢垃圾桶更早被访问。 

population = [greedy_order] + 23 random_orders
for generation in range(35):
    survivors = best_8_by_fitness(population)
    children = crossover_and_mutate(survivors, count=16)
    population = survivors + children
24初始个体：1 个贪心序列 + 23 个随机序列
35迭代代数，持续保留更优访问顺序
0.25变异概率，两点交换避免陷入局部最优
讲述重点：适应度函数不是只看距离，而是 `route_distance + delay_penalty × 0.01`， 因此高满溢点位越晚访问，惩罚越高。

07 / Knowledge Pool

规划历史让实验结果可以复现、比较和解释

每次规划会保存场景快照和规划结果，而不是只保存指标。恢复历史记录时，地图、路线、 AI 上下文都会回到当时状态，便于课堂演示和策略对比。

可复现

保存 seed、threshold、scenario_snapshot、planning_result。

可追溯

保存 fill_history、predictions、events 和 agent trace。

可部署

默认 SQLite fallback，配置后可切换 MySQL 8.4。

可解释

AI 助手读取完整规划上下文，回答路线差异和未分配原因。
